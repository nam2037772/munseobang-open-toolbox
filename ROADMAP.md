# 문서방 5개년 실행 로드맵 (PRODUCT ROADMAP)

> **"네트워크가 차단된 지하 현장에서도 3초 내에 작동하는, 건설인을 위한 무상태 작업 운영체제"**

본 문서는 문서방(Open Toolbox) 프로젝트의 향후 5개년 제품 성장 로드맵 및 구체적 기술적/경험적 실행 전략을 정의합니다. `ARCHITECTURE.md`에 명시된 핵심 수식인 `Workspace = Explorer + Task + Workflow + Workspace` 및 `Local-First` 철학을 절대 훼손하지 않는 것을 원칙으로 합니다.

---

## 1. Vision (비전)

- **한 문장 정의:** 건설 현장의 모든 문서 처리, 기준 조회, 정보 소통을 서버 저장 없이 안전하게 해결하는 **로컬 최초(Local-First) 건설 작업 운영체제.**
- **한 문단 정의:** 문서방은 로그인이나 클라우드 데이터 전송 없이 브라우저 단독으로 실행되는 **무상태 작업대(Stateless Workbench)**입니다. 시공 현장에서 작성되는 수많은 도면 검토, 사진대지, 공사기록을 서버로 탈취하지 않고 온전히 실무자의 로컬 PC에 보전하며, AI와 외부 건설 데이터를 안전하게 중재하는 로컬 허브 쉘로 존재합니다.
- **장기 비전:** 수십만 명의 건설 기술인들이 아침에 출근하여 가장 먼저 실행하는 **"현장 실무 포털"**이자, 오프라인 망분리 현장에서도 100% 독립 실행 가능한 **건설 생태계 표준 OS**가 되는 것입니다.

---

## 2. Product Growth Stage (제품 성장 단계)

### Stage 1: Foundation (OS 쉘 기반 다지기)
- **목표:** 단일 앱들의 원활한 연결 및 극도의 시각적 단순함을 갖춘 OS 쉘 완성.
- **핵심 기능:** 
  - 5대 시간 흐름(루틴) 중심 카테고리 개편.
  - LocalStorage 기반 최근 사용 툴 런처 및 개인화 즐겨찾기 위젯 제공.
  - AI/Google 2분할 과도기 검색창 배치 및 면책 고지 강화.
- **성공 조건:** 신규 카테고리 적용 후 첫 화면 이탈률 20% 감소, 즐겨찾기 도구 진입 속도 3초 미만 달성.

### Stage 2: Workspace (로컬 데이터 버스 연동)
- **목표:** 서버를 거치지 않고 Iframe 앱 간에 데이터가 물 흐르듯 연동되는 로컬 파이프라인 구축.
- **핵심 기능:**
  - `window.parent.postMessage` 프로토콜 기반의 **로컬 데이터 버스(Local Data Bus)** API 배포.
  - 전 단계(예: TBM)의 현장명, 날씨, 작성자가 다음 단계(예: 사진대지, 공사일보)의 문서 서식에 실시간 연동 및 프리필(Pre-fill).
  - 로컬 백업용 표준 JSON 구조 공용화 및 드래그 앤 드롭 이식 구현.
- **성공 조건:** 다단계 공정 도중 사용자의 중복 타이핑 입력 횟수 80% 이상 감소.

### Stage 3: Smart Search (스마트 옴니박스 안착)
- **목표:** 하나의 입력창에서 지능적으로 검색 목적을 파싱하여 결과를 도출하는 통합 검색 경험 완성.
- **핵심 기능:**
  - **스마트 옴니박스(Smart Omnibox)** 단일 입력창 구현.
  - 입력어 유형 분류 엔진(도구 바로가기 / 법령 번호 / AI 실무 질문) 탑재.
  - 검색 결과의 멀티 탭 세션 뷰(Session View) 구현.
- **성공 조건:** 사용자가 검색 카테고리를 선택하기 위한 불필요한 마우스 클릭 횟수 제로(0)화.

### Stage 4: AI Assistant (로컬 및 하이브리드 AI 검토)
- **목표:** 할루시네이션(거짓 답변)이 제어된 신뢰할 수 있는 건설 전용 AI 보조 도구 구축.
- **핵심 기능:**
  - Gemini / GPT / Claude API를 유연하게 스왑하는 실무 `AiSearchProvider` 연동.
  - 사용자가 올린 시공 계획서(로컬 파일)를 서버 전송 없이 브라우저 메모리 안에서 검토하여 KCS 기준과 비교해주는 로컬 AI 검토 모듈(WebLLM / Wasm 기반).
  - 답변에 근거가 되는 KCS/KDS 공식 시방서 원문 구절을 100% 매핑하여 확인시켜주는 원본 링크(Link to Source) 시스템.
- **성공 조건:** AI 검토 의견 중 시방서 일치율 검증 확률 98% 이상 확보.

### Stage 5: Construction OS (오프라인 PWA 및 독립 생태계)
- **목표:** 완벽한 망분리 오프라인 현장 구동성 확보 및 외부 개발사 플러그인 생태계 론칭.
- **핵심 기능:**
  - 전 기능 오프라인 다운타임 제로 작동 PWA(Progressive Web App) 및 로컬 캐싱 엔진.
  - 외부 개발자가 자신만의 툴을 문서방 OS 쉘에 붙일 수 있는 **문서방 SDK 및 플러그인 API** 개방.
  - 오프라인 구동 로컬 LLM(Ollama 연동 혹은 브라우저 내장 모델) 탑재.
- **성공 조건:** 완전 망분리 보안 현장 10곳 이상에서 실무 기본 OS 쉘로 단독 도입 및 외부 서드파티 플러그인 앱 10개 이상 등록.

## 3. App Roadmap (앱 탑재 우선순위)

도구가 무한히 늘어나도 첫 화면의 심플함을 유지하기 위해, 아래의 순서로 순차적 탑재를 진행합니다.

1. **TBM 일지 작성기 (최우선 순위 - 사용 가능):** 매일 아침 필수적인 안전 활동 도구.
2. **사진대지 PDF 작성기 (사용 가능):** 건설 현장 서류의 80%를 차지하는 핵심 사진 정리 도구.
3. **공사일보 작성기 (사용 가능):** 일일 투입 공종 및 인원 기록 도구.
4. **검측프로 (사용 가능):** 시공 확인과 체크리스트 출력을 연동하는 핵심 품질 도구.
5. **계산프로 (사용 가능):** 레미탈 소요량, 구배(기울기), 부가세 환산 등 즉시 계산 도구.
6. **법령 및 국가건설기준(KCS/KDS) 검색기 (신규 - Stage 1):** 최신 시방 규정 상시 조회 도구.
7. **공급원승인 요청서 작성기 (신규 - Stage 2):** 자재 검수와 결합되는 행정 툴.
8. **자재 반입 검수 체크리스트 (신규 - Stage 2):** 콘크리트 및 주요 자재의 현장 반입 품질 관리 도구.
9. **품질시험 기록대장 (신규 - Stage 3):** 공종별 압축강도, 염화물 등 품질 시험 카드 관리.
10. **CAD 기호 및 표준 상세도 자료실 (신규 - Stage 3):** 현장에서 급하게 디테일 도면을 확인하는 경량 뷰어 앱.
11. **AI 시공계획서/시방서 매칭 검토기 (신규 - Stage 4):** PDF/텍스트를 비교하여 규정 위반을 찾아주는 AI 보조 도구.

## 4. Search Roadmap (검색 진화 단계)

```
[v1: 단방향 검색] ────> [v2: 2분할 검색] ────> [v3: 스마트 옴니박스] ────> [v4: 통합 Provider] ────> [v5: 로컬 지식 엔진]
- Google 검색 연동     - AI/Google 분리     - 단일 통합 입력창      - KCS/CSI/뉴스 등       - 오프라인 로컬 LLM
- 외부 탭 이동          - mock 응답 대기     - 입력 즉시 동적 파싱    - 외부 API 멀티캐스트   - 시방서 완전 학습
```

- **v1 (단방향 Google 검색):** 텍스트를 Google Search URL에 매핑하여 새 탭으로 내보내는 단순 형태.
- **v2 (AI/Google 2분할 과도기):** AI 답변 창과 구글 검색창이 수직 배치된 카드 구조. API 키 노출 없이 동작하는 프론트 인터페이스 검증.
- **v3 (Smart Omnibox):** 하나의 입력창에 사용자가 단어를 치면 자동으로 명령어인지, 질문인지 파싱하여 추천 탭 리스트 및 즉시 도구 바로가기 런처 작동.
- **v4 (Search Provider Hub):** 옴니박스 하위에 국토부 보도자료, KCS 건설기준, CSI 사고사례 검색 API를 모듈별로 탑재해 한 곳에서 검색 결과를 수집하는 분산형 쿼리 구조.
- **v5 (Construction Knowledge Engine):** 로컬 저장된 시방 데이터와 오프라인 임베딩 모델을 활용하여, 외부 인터넷망이 완전히 유실되어도 백업 시방서를 조회하여 답을 내는 오프라인 로컬 지식 허브 구축.

## 5. Workspace Roadmap (작업 공간 통합 흐름)

### 1단계: Standalone Sandbox (현재)
- 각 앱들이 브라우저 iframe 내에서 철저히 고립되어 실행됩니다.
- 입력 정보의 계승이 약해, 유저는 현장명, 날씨 등을 매 앱마다 새로 입력해야 합니다.

### 2단계: Connected Workflow (Stage 2)
- 문서방 OS 쉘의 로컬 데이터 버스가 가동됩니다.
- 첫 TBM 단계에서 셋업된 전역 Context(현장 정보, 당일 일자, 작업 내용)가 사진대지와 공사일보로 자동 파이핑(Piping)되어 중복 입력이 완전 제거됩니다.

### 3단계: Unified Construction Workspace (최종)
- 문서방 자체가 하나의 파일 공유 시스템처럼 동작합니다.
- 예컨대, 사진대지 앱 화면에서 드래그한 사진 묶음을 공사일보 iframe이나 검측 점검표 서식 내부로 직접 마우스 드롭할 수 있습니다. 
- 각 앱에서 처리된 임시 JSON 결과물들이 쉘의 사이드바 '임시 보관함'에 보존되어, 사용자는 언제든 여러 개의 임시 서류를 로컬 메모리에 띄워놓고 자유롭게 스위칭하며 최종 PDF 묶음으로 다운로드합니다.

## 6. Design Roadmap (디자인 시스템 진화)

- **UI 진화:** 데스크톱 최적화 레이아웃에서 모바일/패드 야외 사용성으로 진화합니다. 버튼의 터치 타겟은 최소 44px 이상으로 강제하여 현장 조작 편의성을 확보합니다.
- **UX 진화:** 사이드바 메뉴 트리 깊이를 절대로 2단계 이상 늘리지 않습니다. 100개 앱이 도달해도 사용자는 메인 옴니박스를 통해 '자연어 검색'으로 앱을 바로 호출(예: "레미탈 소요량 계산기 켜줘")할 수 있어 탐색 피로도가 평탄하게 유지됩니다.
- **디자인 토큰(MDS v2):** 주간 야외 현장의 높은 조도(태양광 아래)를 고려한 **'아웃도어 고대비(Outdoor High-Contrast) 모드'** 토큰을 신설합니다.
- **컴포넌트 표준화:** 모든 Standalone 앱 개발사들이 동일한 디자인 룩앤필을 가질 수 있도록, 공통 헤더/푸터 및 메타 바용 CSS 스타일 컴포넌트를 패키징하여 오픈소스로 배포합니다.

## 7. Technical Roadmap (기술 스택 전환 계획)

```
[GitHub Pages 정적 배포] ──> [PWA로 오프라인 지원] ──> [Web Messaging Bus API] ──> [로컬 Wasm AI 엔진 이식]
```
1. **GitHub Pages (현재):** 단순 정적 파일 호스팅 및 Client-side 라우팅.
2. **PWA (Progressive Web App):** 서비스 워커(Service Worker)를 등록해 인터넷 단절 시에도 로컬 정적 자원 및 아이콘 캐시를 불러와 다운타임 제로 구동 지원.
3. **Workspace Web Message Bus:** Iframe 간 로컬 통신을 통제하는 `postMessage` 커스텀 쉘 이벤트 핸들러 라이브러리 탑재.
4. **Local Wasm / WebLLM:** AI 질문 기능의 완전한 로컬화를 위해 브라우저 단에서 WebGPU/Wasm을 사용하여 소형 LLM(Llama-3-8B-Instruct 등)을 완전 구동하는 오프라인 엔진 이식.

## 8. Risk Management (피해야 할 전략적 오류)

- **❌ 기능 비대화 및 복잡한 UI (Featuritis):** 모든 요구 서식을 한 화면에 다 때려 박지 않는다. 문서방은 '가장 단순하게 바로 쓰는 작업대'여야 한다.
- **❌ 카테고리 계층 증가:** 카테고리 구조를 3단계 이상으로 깊게 들어가는 디자인은 현장에서 사용을 완전히 기피하게 만든다. 최대 2단계 트리만 허용한다.
- **❌ 회원가입 및 로그인 강제:** 계정 체계를 넣는 순간 문서방 고유의 'Stateless 로컬 보안' 신뢰성이 붕괴된다. 개인화는 오직 브라우저 내부 LocalStorage에만 귀속시킨다.
- **❌ 클라우드 서버 저장소 지향:** 사용자 데이터 및 작업 서류를 우리 서버로 전송받아 DB화하려는 충동을 통제한다. 이 사상을 잃으면 엔터프라이즈(건설 대기업, 군 보안 공사) 시장의 신뢰를 잃는다.

## 9. Success Metrics (성공 측정 지표)

- **⏱️ 3초 룰 (Accessibility):** 첫 대시보드 화면에 접속해 원하는 도구 카드나 기능을 찾아 입력 포커스를 띄우는 데 드는 시간 3초 이내.
- **⏱️ 30초 룰 (Efficiency):** 옴니박스나 사이드바에서 도구를 실행하고 로컬 메모리에 첫 자를 입력하는 데 걸리는 시간 30초 이내.
- **📊 100+ 앱 스케일 생존율:** 도구의 수가 100개가 되어도 대시보드 메인 첫 화면이 복잡해지지 않고, 사이드바 트리가 화면 높이를 벗어나 오버플로우되지 않도록 필터링이 정상 가동되는지 여부.
- **🔒 데이터 유출 제로:** 외부 서버 통신 감사 도중 시공 정보나 사용자의 개인정보가 담긴 텍스트가 네트워크 탭에 잡히지 않을 확률 100%.

## 10. Priority Matrix (우선순위 매트릭스)

### 🟢 반드시 해야 하는 것 (Must Do)
- 스마트 옴니박스 단일 통합 검색창 구현.
