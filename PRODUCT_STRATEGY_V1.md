# [제품 전략 보고서] 문서방 PRODUCT STRATEGY V1

> **"웹사이트를 넘어, 건설인의 로컬 보안 작업 운영체제(Construction OS)로"**

본 문서는 문서방(Open Toolbox)을 단순한 정적 웹 허브에서 건설인이 아침에 출근하여 가장 먼저 실행하는 **로컬 최초(Local-First) 건설 작업 운영체제(OS)**로 성장시키기 위한 장기 제품 전략 명세서입니다. 기존 `ARCHITECTURE.md`와 `MUNSEOBANG_FRONT_V3_DIRECTION.md`에 규정된 핵심 가치를 훼손하지 않고, 이를 5개년 로드맵으로 확장하여 구체화합니다.

---

## 1. Executive Summary

문서방은 **"서버 저장소가 아닌, 무상태 작업대(Stateless Workbench)"**라는 고유한 사상을 가진 유일무이한 건설 실무 플랫폼입니다. 
본 전략은 문서방을 단순한 도구 링크 저장소에서 **'통합 건설 작업 쉘(Construction OS Shell)'**로 진화시키는 것을 목표로 합니다. 
- **핵심 목표:** 건설인이 출근해서 퇴근할 때까지 브라우저 탭 하나로 모든 문서 작성, 기준 조회, 사진 정리를 해결하며 로컬 보안을 100% 유지하는 것.
- **핵심 전략:** 
  1. 단일 검색 인터페이스인 **스마트 옴니박스(Smart Omnibox)**를 통한 인지 부하 감소.
  2. 단절된 웹앱들을 연결하는 **로컬 데이터 버스(Local Data Bus)** 및 워크스페이스 구축.
  3. 도구의 개수가 100개 이상으로 늘어나도 미니멀리즘을 유지하는 **공정/흐름 기반 정보 구조(IA)** 개편.

---

## 2. Current State Analysis

현재 문서방은 정적 쉘 프레임에 독립 웹앱(사진대지, 공사일보, 계산기 등)을 iframe으로 임베딩하여 연동하는 형태입니다. 
- **프론트엔드 역할:** 2단 레이아웃(사이드바 + 메인 캔버스)을 갖추고 있으며, 사이드바에서 업무를 선택하면 타임라인에 따라 도구가 캔버스에 로딩됩니다.
- **데이터 흐름:** 독립 앱들이 각각 독립적으로 동작하며, 현장명/날씨 등의 기본 컨텍스트만 상위 Context를 통해 쿼리 파라미터로 넘겨주는 단순 구조입니다.
- **진단:** 로컬 우선주의와 샌드박스 안정성은 확보했으나, 도구 간의 완성도 편차가 존재하고, 신규 도구들이 추가될 때 대시보드가 파편화될 임계점에 도달해 있습니다.

---

## 3. Strengths (강점 분석)

1. **완벽한 로컬 보안 (Zero-Server Vulnerability):**
   서버에 유저 데이터를 저장하지 않으므로, 정보 유출에 극도로 예민한 건설 현장에서 발주처/감리단/시공사 모두 안심하고 사용할 수 있습니다.
2. **독립 실행력 (Pluggable Micro-Frontends):**
   개별 툴(사진대지프로, 검측프로 등)이 독립된 URL로 동작하므로, 개별 앱의 장애가 전체 OS 쉘로 전파되지 않으며 개별 업데이트가 자유롭습니다.
3. **MDS의 다크모드 최적화:**
   빛 반사가 심한 주간 현장 및 먼지가 많은 현장 상황을 고려해 고대비/다크모드가 완성도 있게 디자인되어 있습니다.
4. **점진적 노출 (Progressive Disclosure) 설계:**
   처음부터 복잡한 서식을 주지 않고 `폴더 -> 태스크 -> 도구`로 이어지는 레이어를 두어 유저 진입 장벽이 낮습니다.

---

## 4. Weaknesses (약점 분석)

1. **검색 인터페이스의 분절화:**
   AI 질문창과 Google 검색창이 나열식으로 존재해 유저가 검색 전 '선택 단계'를 거쳐야 함.
2. **도구 간 시퀀스 전달의 한계:**
   Iframe 간 데이터 이동이 쿼리 스트링 방식에 제한되어 있어, 사진대지에서 생성한 PDF나 검측 서류를 공사일보에 직접 드래그 앤 드롭하는 식의 복잡한 로컬 자원 연동이 불가능함.
3. **확장 불가능한 내비게이션 구조:**
   사이드바 트리에 모든 도구를 일렬로 나열하는 방식은 도구가 30개를 초과할 경우 스크롤 병목 및 메뉴 탐색 실패율이 급증함.
4. **개인화의 부재:**
   Stateless 철학으로 인해 로그인 정보가 없다 보니, 사용자가 매일 반복해서 사용하는 '나만의 루틴 도구'를 첫 화면에 즉시 띄워주는 지능형 런처 기능이 약함.

---

## 5. Future Vision

> **"건설 현장의 Microsoft Windows가 되는 것"**

문서방은 단순한 링크 허브를 넘어, 건설 현장 실무자가 브라우저를 켜자마자 띄우고 퇴근할 때 닫는 **Web-based Local OS**가 됩니다.
- **1년 내:** 스마트 옴니박스를 통한 통합 검색 체계 안착 및 로컬 임시 개인화 환경 구축.
- **3년 내:** 로컬 웹 공유 버스(Web Message Protocol)를 고도화하여 도구 간에 대용량 파일(PDF, 이미지) 및 JSON 데이터를 드래그 앤 드롭으로 주고받는 **Inter-App Data Flow** 완성.
- **5년 내:** 오프라인 동작(PWA) 지원을 통한 완전한 망분리 현장 대응 및 로컬 LLM(Ollama 등) 브라우저 내 연동을 통한 오프라인 AI 작업 환경 완성.

## 6. Information Architecture (IA)

도구가 100개 이상으로 증식해도 뇌에 무리를 주지 않기 위해, **건설인의 하루 업무 루틴(시간 흐름)**에 맞춘 **5대 마스터 카테고리**로 계층을 단일화합니다.

```
[문서방 OS 루트]
├── 1. 현장 안전 TBM (아침 업무)
│   ├── 일일 TBM 일지 작성
│   ├── 유해위험요인 발굴 (위험성평가)
│   └── 안전교육 및 서명 대장
├── 2. 시공 & 사진 (주중 현장 업무)
│   ├── 사진대지 PDF 작성기
│   └── 일일 공사일보 작성
├── 3. 검수 & 검측 (시공 후 확인 업무)
│   ├── 검측프로 (검측서/체크리스트)
│   ├── 자재 반입 및 검수 체크
│   └── 자재 공급원 승인 요청
├── 4. 행정 & 공무 (본사 및 계약 업무)
│   ├── 설계변경 및 기성 청구
│   ├── 공문/서한 작성기
│   └── 준공 검사 제출 서류 점검
└── 5. 기준 & 정보 (상시 조회 업무)
    ├── KCS / KDS 기준 검색
    └── 실무 건설 뉴스 브리핑
```

## 7. Navigation Strategy

도구 개수 스케일링 문제를 해결하기 위한 구체적인 내비게이션 전략을 제안합니다.

### ① 2단계 계층 구조의 절대적 사수
- 내비게이션의 깊이(Depth)는 **`대분류 폴더 (1단계) > 세부 태스크 (2단계)`**로 완전히 고정합니다. 그 이하의 복잡성은 태스크 내부의 `Workflow Node`를 통해 점진적으로 노출합니다.

### ② VS Code 스타일의 액티비티 바(Activity Bar) 도입
- 사이드바를 단순히 접고 펴는 것을 넘어, 가장 좌측에 극도로 얇은 아이콘 바(Activity Bar)를 배치합니다.
- 아이콘 구성: `[홈]`, `[최근 사용(Recents)]`, `[즐겨찾기(Favorites)]`, `[도구 탐색기(Explorer)]`, `[설정]`
- 이를 통해 사이드바가 닫혀 있는 상태에서도 주요 업무 영역 간 스위칭을 단 1회의 탭으로 수행합니다.

## 8. Search Strategy (스마트 옴니박스 설계)

사용자는 AI 질문창, 구글 검색창, 도구 검색창을 따로 쓰지 않고 단 하나의 **스마트 옴니박스(Smart Omnibox)**에 검색합니다.

```
┌────────────────────────────────────────────────────────┐
│  ✦ 무엇을 도와드릴까요? (예: 되메우기 기준, 사진대지)     │
└────────────────────────────────────────────────────────┘
  ↓ [입력 즉시 동적 쿼리 분기 파싱]
  ├─ 1. "사진대지" 입력시 ──────> 자동완성: [바로가기] 사진대지 PDF 작성기 실행
  ├─ 2. "되메우기 기준" 입력시 ──> KCS/KDS API 매칭 결과 리스트 노출
  └─ 3. "단열재 보호해야해?" ──> [AI 답변] 탭 선택 유도 (Mock/LLM 연동)
```

### ① 로컬 파서(Parser) 및 세션 오케스트레이션
- 옴니박스에 텍스트가 입력되면 내장된 경량 정규식 엔진이 쿼리를 분석합니다.
  - 명령어/도구 키워드가 포함된 경우: **내부 런처 프로바이더** 작동.
  - 시방서 번호(KCS 11 20 20 등)나 전문 용어가 포함된 경우: **건설기준 프로바이더(KCS/KDS)** 작동.
  - 질문형 종결어(~야?, ~인가요?, ~추천)가 감지되는 경우: **AI 답변 프로바이더**를 기본 결과로 바인딩.
- 검색 결과 피드백 영역은 **세션 뷰(Session View)** 형태로 구성하여, 한 화면 내에서 `[AI 답변]` / `[건설 기준]` / `[구글 검색(새탭)]` 탭을 슬라이딩하여 비교 확인할 수 있도록 합니다.

## 9. Workspace Strategy (로컬 데이터 버스)

문서방 내부에서 여러 독립 웹앱들이 유기적으로 데이터를 주고받기 위한 **Workspace 통신 규약**을 정의합니다.

### ① Shared Context 프로토콜 (로컬 데이터 버스)
- 문서방 쉘(Shell)은 중앙에서 `MemoryContext`를 관리하며, 개별 iframe 앱은 `window.parent.postMessage` API를 통해 데이터를 갱신하거나 타 도구의 데이터를 요청합니다.
- **예시 흐름 (콘크리트 타설 공정):**
  1. `TBM 일지 작성기`에서 입력된 '현장명: 제주 아라동', '일자: 2026-07-08' 정보가 로컬 세션에 기록됨.
  2. 다음 단계인 `사진대지 PDF 작성기`로 이동하면, 사진대지 앱이 로컬 세션에서 현장명과 일자를 읽어와 앨범 표지에 자동으로 프리필(Pre-fill)함.
  3. 마지막 `검측프로`로 진입하면, 사진대지에 업로드했던 사진 목록과 날씨 정보가 자동으로 검측요청서 파일 첨부 영역 및 서식에 매핑됨.
- 이 과정에서 서버는 일절 개입하지 않으며, 오직 브라우저 탭의 전역 메모리 상에서 `postMessage` 이벤트 리스너를 통해 구현됩니다.

## 10. Design System (MDS 고도화)

문서방 디자인 시스템(MDS)은 단순한 CSS 변수 모음을 넘어, **Standalone App 개발사들이 그대로 가져다 쓸 수 있는 UI 키트(Kit)** 형태로 배포되어야 합니다.

### ① 디자인 토큰 고도화 (MDS v2)
- **야외 고대비 모드(Outdoor High-Contrast):** 주간 야외 현장에서 태양광 아래 패드/폰 화면이 안 보이는 현상을 막기 위해, 명도 대비 7:1 이상의 고대비 테마 변수를 추가로 지정합니다.
- **인체공학적 터치 그리드(44px Rule):** 현장 작업자가 두꺼운 장갑을 끼거나 먼지 묻은 손으로 조작할 때 오작동을 차단하기 위해, 모든 대시보드 버튼과 검색창 높이를 최소 44px 이상으로 강제 규정합니다.
- **상태 카드 컴포넌트(State Cards):** 독립 앱들의 카드 형태를 통일하기 위해 `.mds-card--active`, `.mds-card--pending`, `.mds-card--disabled` 클래스의 스펙을 공통 CSS로 배포합니다.

## 11. Roadmap (5개년 발전 계획)

### 🚀 Phase 1: OS 기초 체력 확보 (1년 차 - v2.5 ~ v3.0)
- 스마트 옴니박스(Smart Omnibox) 단일 검색 인터페이스 개발 및 AI Provider 연동.
- LocalStorage 및 쿠키 기반의 '로컬 자동 저장 및 최근 사용 도구' 개인화 기능 개발.
- 5대 업무 흐름 중심 IA 개편 및 사이드바 액티비티 바 UI 도입.

### 🔌 Phase 2: 연결성 극대화 (2~3년 차 - v4.0)
- Iframe 간 대용량 데이터 전송을 지원하는 **로컬 데이터 버스(Local Web Messaging Bus)** 표준 규약 배포.
- 타 앱에서 내보낸 JSON 데이터를 드래그 앤 드롭만으로 파싱해가는 데이터 임포트 엔진 구축.
- 건설기준센터(KCS/KDS) 공식 오픈 API를 로컬에서 직접 쿼리하는 클라이언트 검색 모듈 개발.

### 🌐 Phase 3: 오프라인 OS 및 생태계 구축 (4~5년 차 - v5.0)
- PWA(Progressive Web App) 기술을 전면 적용하여 인터넷이 끊긴 산간/터널 현장에서도 100% 구동되는 오프라인 툴박스 완성.
- 브라우저 내 오프라인 인퍼런스가 가능한 경량 로컬 LLM(WebLLM / Wasm 기반 모델) 옴니박스 내 기본 탑재.
- 외부 개발자가 문서방 SDK를 다운받아 자사 앱을 문서방 생태계(OS)에 쉽게 플러그인할 수 있는 개방형 배포 채널 오픈.

## 13. 문서방만의 차별점 (Unique Selling Proposition)

- **망분리 현장 최적화 (Air-gapped Field Ready):**
  군부대 공사, 정부 기관 보안 현장, 지하 터널 등 외부 네트워크가 철저히 차단되거나 보안이 극도로 엄격한 장소에서 유일하게 정상 구동 및 보안 심의를 패스할 수 있는 전천후 로컬 작업대입니다.
- **다운타임 제로 (Zero Downtime):**
  서버 장애나 트래픽 초과로 시스템 전체가 마비되는 일반 클라우드 서비스와 달리, 프론트엔드 정적 파일이 로컬 캐시에 등록된 순간부터 서버 통신 없이 언제든 열어 작업할 수 있습니다.
- **초고속 반응성 (Instant UI):**
  데이터 전송 레이턴시가 전혀 없어 입력, 계산, PDF 출력이 실시간 0ms로 이루어집니다.

## 14. 우선순위가 높은 개선사항 TOP 20 (Action Items)

1. **[Search] 단일 스마트 옴니박스 UI 적용:** AI 및 구글 검색창을 물리적으로 1개로 통일.
2. **[Architecture] 통합 `SearchProvider` 클래스 작성:** 각 검색 채널(AI/Google/KCS/도구)을 동적으로 중재하는 프로바이더 추상화.
3. **[IA] 공정 흐름 기준 5대 대분류 개편:** 안전/시공/품질/공무/참조의 생애주기 카테고리화.
4. **[Navigation] 좌측 슬림 액티비티 바(Activity Bar) 구현:** 사이드바 공간 최적화 및 탭 이동성 개선.
5. **[UX] 44px 장갑 터치 타겟 가이드라인 제정:** 모바일 전용 대형 버튼 템플릿화.
6. **[UX] 로컬스토리지 기반 최근 사용 도구 위젯:** 대시보드 2단 영역에 퀵 런처로 배치.
7. **[Aesthetics] 실외 고대비 다크 테마 추가:** 야외 시인성 개선용 고대비 토큰 배포.
8. **[Workspace] Iframe postMessage 로컬 통신 라이브러리 개발:** 앱 간 데이터 공유 브릿지 구축.
9. **[UX] 추천 검색 태그(Quick Search Chips) 기능:** 입력창 밑에 추천 질문 칩 제공.
10. **[Stateless] 무상태 로컬 임시저장(Autosave) 안전장치:** LocalStorage 캐시를 최대 50MB 한도로 자동 세션 유지.
11. **[Design] MDS v2 Typography 세부 조정:** 영문/숫자 혼용이 많은 시방서 가독성을 위해 Noto Sans KR/Montserrat 자간 미세 조정.
12. **[Search] KCS/KDS 공식 오픈 API 검색 모듈 연구:** 클라이언트 단독 API 연동 검색 기능 확보.
13. **[Workspace] 도구 간 드래그 앤 드롭 파일 공유 기능:** 사진대지의 결과 PDF를 검측프로 서식에 직접 드롭 가능하도록 쉘 영역 이벤트 제어.
14. **[UI/UX] 옴니박스 자동완성 리스트 레이어:** 도구 이름 입력 시 즉각 실행 가능한 추천 검색창 드롭다운 렌더링.
15. **[Brand] "문서방" 로컬 OS 브랜드 아이덴티티 수립:** "우리는 정보를 모으지 않고, 당신의 로컬 PC에 정답만을 정리합니다." 카피 히어로 노출.
16. **[Quality] 독립 실행형 앱의 쉘 감지 기능:** 앱이 단독 구동 시와 쉘 임베디드 구동 시를 자동 구분해 메타 바 노출 제어.
17. **[Architecture] PWA 서비스 워커(Service Worker) 도입:** 완벽한 오프라인 작업 환경을 위한 리소스 캐싱 프레임워크 셋업.
18. **[UI] 옴니박스 우측 멀티 액션 버튼 적용:** 엔터 전 `[AI 질문]` / `[건설기준]` 선택형 시각 단추 노출.
19. **[Security] 로컬 세션 데이터 '즉시 영구 삭제' 단추 구현:** 사용 완료 후 1클릭으로 브라우저 메모리와 캐시를 청소하는 보안 파쇄기 기능 제공.
20. **[Collaboration] 로컬 JSON 내보내기/가져오기 표준 템플릿화:** 사용자가 파일 한 장만 동료에게 이메일로 보내면 즉각 협업이 이어지도록 구조 표준화.

## 15. Conclusion

문서방은 단순한 웹 도구 모음이 아닙니다. **"클라우드가 지배하는 시대에, 데이터의 완전한 개인 소유권과 오프라인 연속성을 보장하는 가장 진보된 건설인용 보안 운영체제"**입니다. 
본 전략 보고서의 설계안대로 스마트 옴니박스를 중심으로 한 검색 통합, 로컬 데이터 버스를 통한 도구 간 연동, 그리고 시간 흐름 기반의 IA를 구축한다면, 문서방은 향후 5년 내 수십만 명의 건설 실무자가 아침에 가장 먼저 켜고 종일 머무는 독보적인 실무 플랫폼으로 자리매김할 것입니다.
