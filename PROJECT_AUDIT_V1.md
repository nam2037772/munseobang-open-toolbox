# 프로젝트 설계 정합성 감사 보고서 (PROJECT AUDIT V1)

> **"오버엔지니어링(Over-engineering) 방지와 단순성 회복을 위한 아키텍처 비평"**

본 감사는 문서방(Open Toolbox) 프로젝트 내 핵심 설계 자산들(`ARCHITECTURE.md`, `MUNSEOBANG_FRONT_V3_DIRECTION.md`, `PRODUCT_STRATEGY_V1.md`, `ROADMAP.md`) 간의 철학적·구조적 충돌을 진단하고, YAGNI(You Aren't Gonna Need It) 원칙에 입각하여 과도하게 설계된 기술 요소를 지적함으로써 프로젝트 본연의 현실적인 궤도를 확보하는 데 목적이 있습니다.

---

## 1. 철학 충돌 검토 (Philosophical Consistency)

네 개의 문서 모두 `Local-First`와 `Stateless Workbench`를 기본 가치로 천명하고 있으나, 세부 발전 계획으로 진입하면서 아래와 같은 철학적 자기모순 및 충돌 징후가 식별되었습니다.

- **무상태(Stateless) 철학과 로컬 데이터 스토리지의 충돌:**
  - `ARCHITECTURE.md`에서는 "새로고침되거나 브라우저 탭이 닫히면 데이터가 휘발되는 상태 없는 작업대"를 선언했습니다.
  - 그러나 `ROADMAP.md` Stage 5 및 `PRODUCT_STRATEGY_V1.md` 21번 과제에서는 "오프라인 모드를 위해 로컬 데이터베이스(IndexedDB) 구조 연계"를 제안했습니다.
  - **진단:** 로컬 브라우저에 IndexedDB 수준의 영구 데이터베이스를 구축하고 상태를 복잡하게 관리하기 시작하면, 싱크 상태 및 브라우저 캐시 보존율에 따라 동기화 오류와 정합성 문제가 발생하며, 무상태가 제공하던 **"새로고침하면 깨끗이 해결된다"는 무결성 가치**가 훼손됩니다.
- **AI 보조자 위상과 로컬 LLM 이식의 주객전도:**
  - `MUNSEOBANG_FRONT_V3_DIRECTION.md`는 "AI는 판단자가 아닌 실무 보조자"라고 한정 지었습니다.
  - 하지만 `ROADMAP.md` 후반부에서는 "Wasm 기반 WebLLM의 브라우저 내 단독 실행 및 Ollama 로컬 구동"을 핵심 로드맵으로 삼고 있습니다.
  - **진단:** 모바일/태블릿 등 현장 저사양 환경에서 수십억 파라미터의 로컬 LLM을 브라우저 런타임에 올리는 것은 리소스 폭발(Crash)을 초래하며, 보조 역할에 불과한 AI를 위해 엄청난 로컬 자원을 낭비하는 과도한 엔지니어링입니다.

## 2. 구조 및 정의 충돌 (Structural Conflict)

각 문서에서 동일 대상을 서로 상이한 수준이나 방식으로 정의한 부분들이 발견되었습니다.

- **카테고리(IA)의 명칭 및 구성 불일치:**
  - `PRODUCT_STRATEGY_V1.md`에서는 **5대 카테고리**로 개편안을 명확히 고정 제안했습니다.
    *(1. 현장 안전 TBM / 2. 시공 & 사진 / 3. 검수 & 검측 / 4. 행정 & 공무 / 5. 기준 & 정보)*
  - 그러나 `apps.ts` 실 데이터 파일 및 `ROADMAP.md`의 과제 목록에서는 여전히 기존의 4대 분류 또는 다른 카테고리 네이밍이 혼재되어 표기되어 있습니다. (예: '안전·환경', '공무·행정' 등 기호 사용 차이)
- **도구 찾기(Ctrl+K) 검색창과 Smart Omnibox의 경계 불명확:**
  - `App.tsx` 및 `CommandBar.tsx`에서는 전역 핫키 `Ctrl + K`를 이용한 별도의 팝업 검색창을 띄우는 형태로 도구 검색을 정의하고 있습니다.
  - 반면 `MUNSEOBANG_FRONT_V3_DIRECTION.md`와 `PRODUCT_STRATEGY_V1.md`에서는 단일 옴니박스가 켜지면 내부 도구까지 즉각 런처 형태로 연결된다고 선언해, **두 개의 다른 검색 상태가 동일 화면 내에서 어떻게 인터랙티브하게 충돌을 방지할지** 정의하지 않았습니다.

## 3. 중복 및 유지보수 비용 (Redundancy Audit)

- **제품 전략(PRODUCT_STRATEGY_V1.md)과 실행 로드맵(ROADMAP.md)의 난립:**
  - 두 문서 모두 `1년 / 3년 / 5년` 단위의 성장 단계와 로드맵을 정의하고 있습니다.
  - 특히 전략 문서의 `Top 20 Recommendations`와 로드맵 문서의 `Top 30 실행 과제`는 약 70% 이상의 내용이 완전히 중복되거나 명칭만 미세하게 다릅니다.
  - **진단:** 향후 로드맵의 방향성이 수정될 경우, 두 개의 거대한 기획 문서를 동시에 갱신해야 하므로 **문서 파편화 및 갱신 비용**이 매우 커지게 됩니다.

## 4. 누락된 규칙 (Missing Rules)

현재 설계 문서들에는 다음의 중대한 프로젝트 생태계 규칙이 빠져 있습니다.

- **개별 독립 웹앱(Micro-Frontend)의 API 스펙 표준 문서 누락:**
  - `ARCHITECTURE.md`에서 각 도구(사진대지 등)는 독립 앱(Standalone)이고 `postMessage`로 연결된다고 했으나, **정확히 어떤 JSON 필드명과 타입 스펙으로 상호 합의할지**에 대한 최소한의 인터페이스 규격서가 부재합니다. 이 규격이 없으면 외부 개발사가 플러그인을 제작하는 것은 불가능합니다.
- **오프라인 환경에서의 '로컬 파일 복구' 라이프사이클 정책 누락:**
  - 무상태 환경에서 작성하던 서류를 잃어버리지 않게 해주는 최소한의 로컬 스토리지 라이프사이클(예: 세션 만료 시간, 자동 삭제 임계 용량)에 대한 기획적 합의가 누락되었습니다.

## 5. 현실성 및 YAGNI 비평 (YAGNI & Feasibility Critique)

5개년 로드맵 중 아래 항목들은 **YAGNI(You Aren't Gonna Need It: 정말 필요하기 전엔 만들지 말라)** 원칙에 위배되거나 현실성이 매우 떨어집니다.

1. **브라우저 내 WebLLM/로컬 LLM 이식 (Stage 4~5):**
   - 앞서 언급한 바와 같이, 모바일 크롬 브라우저 상에서 Wasm/WebGPU 기반으로 LLM을 로드하는 것은 메모리 과부하로 인한 강제 탭 종료를 일으킵니다. 건설인은 대규모 AI 학습기가 아니라 단지 "되메우기 전에 단열재 보호해야 해?"에 대한 **시방서 기반의 가벼운 RAG 검색 결과**를 3초 안에 보고 싶어 할 뿐입니다. AI 서버 API에 쿼리를 던지고 텍스트만 리턴받는 클라이언트 구조면 충분합니다.
2. **독립 플러그인용 개발자 SDK 및 npm 배포 (Stage 5):**
   - 문서방은 상업적 App Store 플랫폼이 아닙니다. 비정기적으로 추가되는 서너 개의 자매 앱들을 위해 거대한 SDK 패키지를 만들고 npm에 퍼블리싱하는 것은 과도한 플랫폼 지향형 오버엔지니어링입니다. 단순한 공용 CSS 파일 한 장과 데이터 바인딩 가이드 문서 한 장이면 해결됩니다.
3. **영구 로컬 DB(IndexedDB)의 전면 연동 (Stage 5):**
   - 무상태 작업대 철학에 반하는 과도한 저장 레이어입니다. 사용자가 작업 도중 `내보내기` 단추를 눌러 `.json` 파일로 직접 PC에 저장하고, 다음에 이어서 쓸 때 그 파일을 `화면에 드롭`하여 복원하게 만드는 것이 가장 안전하고 아키텍처를 깔끔하게 유지하는 비결입니다.

## 6. 향후 6개월 최우선 행동 과제 TOP 10 (6-Month Action Items)

오버엔지니어링을 배제하고 가장 기본적이면서 사용자 가치가 높은 항목 10개만 집중 추려냅니다.

1. **[Search] 옴니박스 단일 마크업 통합 및 UI 1단계 개편:** 복잡한 파서 없이 AI(임시 결과)와 Google 검색을 하나의 인풋 필드에서 엔터 전 버튼 분기로 선택 실행하는 UI 안착.
2. **[IA] apps.ts 카테고리 5대 흐름 재매핑 적용:** 왼쪽 사이드바 트리를 5대 카테고리로 갱신.
3. **[Workspace] 쿼리 스트링(Query String)을 통한 1단계 데이터 계승 규격화:** 현장명, 날씨, 날짜 정보를 Iframe 로드 시 URL 파라미터로 안전하게 넘겨주는 로직 통일.
4. **[UX] 대시보드 내 LocalStorage 기반 최근 사용 도구 위젯 구현:** 최근 실행한 도구 3개를 쿠키/로컬 스토리지에 세션 유지하여 대시보드 최상단 노출.
5. **[UI] 44px 모바일 터치 타겟 크기 표준화:** 모바일 디바이스 뷰포트에서 버튼 클릭 미스 방지.
6. **[Policy] AI 검색결과 면책 고지 안내창 상시 노출 적용:** 법적 리스크 사전 차단.
7. **[Search] 국토부/CSI 검색을 대체할 Google Custom Search Engine(CSE) 연동:** 복잡한 외부 기관 API를 일일이 연동하는 대신, 구글 맞춤검색 API 한 줄로 국토부/건설기준 영역을 필터링해 가져오는 초경량 검색 연동.
8. **[UI] 사이드바 좌측 VS Code형 미니멀 액티비티 바(아이콘바) 구현:** 화면 공간 확보.
9. **[Security] 1클릭 브라우저 임시 스토리지 영구 파쇄 기능 구현:** 군부대 등 보안 현장용 즉시 파쇄 단추.
10. **[Quality] 독립 앱 iframe의 Sandbox 보안 속성 최적화:** `allow-scripts allow-same-origin` 등 필요한 최소 권한만 설정하여 보안성 유지.

## 7. 문서 구조 개선 제안 (Document Restructuring)

설계 문서의 비대화를 방지하고 관리 효율을 극대화하기 위해 다음과 같이 문서를 정리할 것을 제안합니다.

- **[통합] DIRECTION + STRATEGY + ROADMAP ──> `PRODUCT_MANIFESTO.md` 로 통합 단일화:**
  - 현재 이 세 문서는 비전, 로드맵, 액션 아이템이 도처에 산재해 있어 일관성 있게 관리하기 불가능에 가깝습니다.
  - `MUNSEOBANG_FRONT_V3_DIRECTION.md`를 삭제하고, `PRODUCT_STRATEGY_V1.md`와 `ROADMAP.md`에서 겹치는 로드맵/개선과제를 쳐낸 후, **단 하나의 제품 명세서인 `PRODUCT_MANIFESTO.md`**로 병합할 것을 권고합니다.
- **[유지] `ARCHITECTURE.md`:**
  - 핵심 개발 공식과 Local-First, Standalone 등의 설계 원칙을 담고 있으므로 단독으로 명확히 유지하되, Wasm LLM이나 IndexedDB 같은 과도한 확장 시도는 이 문서에서 철저히 차단해야 합니다.

## 9. 결언 (Conclusion)

문서방의 힘은 **"더 많은 기술"이 아니라 "더 철저한 비워내기"**에서 나옵니다. 
현실성이 결여된 브라우저 구동형 로컬 LLM이나 IndexedDB 관리층을 걷어내고, 초심으로 돌아가 **단일 스마트 옴니박스**와 **Iframe 데이터 파이프라인(Message Bus)**이라는 두 가지 핵심 가치에만 6개월의 개발 리소스를 온전히 쏟아붓는 것이 성공을 위한 가장 확실한 전략입니다.
