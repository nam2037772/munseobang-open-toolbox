# ARCHITECTURE.md
> **문서방 Construction OS 설계 원칙 및 시스템 아키텍처**

본 문서는 문서방(Open Toolbox) 프로젝트의 핵심 설계 철학과 기술적 가이드라인을 규정하는 아키텍처 표준 명세서입니다. 이 프로젝트에 참여하는 모든 개발자, 디자이너 및 AI 에이전트는 본 문서에 정의된 원칙을 절대적으로 준수해야 하며, 본 원칙을 훼손하는 형태의 코딩 및 설계는 허용되지 않습니다.

---

## 1. 핵심 아키텍처 공식 (Core Architecture Formula)

문서방의 모든 인터페이스와 사용자 경험은 아래 공식에 의해서만 구성됩니다.

$$\text{Construction Workspace} = \text{Explorer} + \text{Task} + \text{Workflow} + \text{Workspace}$$

-  **Explorer (탐색기)**: 윈도우 탐색기처럼 친숙한 대분류 폴더 트리로 전체 공정의 범위를 공간적으로 이해시키는 맵입니다.
-  **Task (업무)**: 폴더 내부에 위치하며, 사용자가 실제 수행해야 하는 구체적인 실무 과제입니다 (예: 콘크리트 타설).
-  **Workflow (흐름)**: 태스크를 선택했을 때 펼쳐지는 도구 간의 시각적 연결 체인(Node Chain)입니다.
-  **Workspace (작업대)**: 흐름의 마디를 클릭하여 실제 문서를 작성하고 다운로드하는 캔버스 에디터 영역입니다.

---

## 2. 절대 변경 불가 설계 원칙 (Immutable Architectural Principles)

### 2.1 Local-First (로컬 우선 원칙)
-  **서버 저장 없음**: 문서방은 클라우드 데이터베이스에 사용자의 문서를 수집하거나 저장하지 않습니다.
-  **샌드박스 연산**: 모든 데이터 입력, 이미지 드래그 앤 드롭, PDF 렌더링, 계산 연산은 서버 호출 없이 오직 사용자 브라우저 메모리 내부에서 처리됩니다.
-  **로컬 백업**: 사용자가 작성한 결과물은 사용자의 로컬 PC 파일시스템(`.json`, `.pdf`, `.zip`)에만 저장되며, 이어 쓰기를 위한 백업 파일 가져오기 또한 로컬 파일을 브라우저에 드롭하는 방식으로 구동됩니다.

### 2.2 Progressive Disclosure (점진적 노출)
-  **인지 과부하 차단**: 처음 진입 시에는 모든 도구와 폼을 노출하지 않고 극단적으로 단순한 상위 폴더 트리만 보여줍니다.
-  **경로 기반의 해제**: 사용자가 `Explorer`에서 경로를 선택하고 깊이 진입할수록 `Workflow`가 가지를 치며 빛나고, 단계를 클릭해야 비로소 `Workspace` 입력창이 순차적으로 노출됩니다.

### 2.3 MDS (Munseobang Design System) 절대 준수
-  **디자인 일관성**: 모든 CSS 및 React 컴포넌트는 MDS 토큰(`04_DESIGN_TOKEN.md`)에 정의된 HSL 색상, 8px Spacing Grid, Radius, Typography 스펙만을 사용합니다. 하드코딩된 인라인 스타일링은 엄격히 차단됩니다.

### 2.4 Stateless Workbench (상태 없는 작업대)
-  문서방의 프론트엔드 프레임은 영구적인 백엔드 상태를 갖지 않습니다.
-  오직 사용자가 브라우저 탭을 열어 작업하고 있는 동안의 임시 전역 상태(Context)만 존재하며, 브라우저가 새로고침되거나 닫히면 메모리 내 데이터는 소멸합니다. (필요 시 로컬 스토리지에 한해 최소한의 자동 저장만 지원합니다.)

### 2.5 독립 실행 가능한 애플리케이션 (Standalone Tools)
-  **모듈화**: 문서방에 들어오는 개별 앱(사진대지프로, 공사일보프로, 검측프로 등)은 문서방 프레임이 없이도 단독 웹 브라우저에서 실행 및 배포될 수 있어야 합니다.
-  **플러그형 설계**: 각 앱은 독립적인 HTML/JS 파일이나 서브 도메인 형태로 배포되며, 문서방 쉘(Shell)과는 표준화된 API(예: Iframe postMessage 또는 Local State Props)를 통해서만 데이터를 주고받습니다.

### 2.6 Front의 OS 허브 역할 (Front as a Shell OS)
-  문서방 프론트엔드는 도구를 개발하는 곳이 아닌, 분리된 도구들을 **연결하고 경로를 찾아주며 데이터를 연동해 주는 OS 커널 및 쉘(Shell) 역할**에 집중합니다.
-  허브는 1단계 도구의 완료 이벤트를 가로채어 2단계 도구의 입력 폼에 프리필(Pre-fill) 데이터를 매핑해 주는 브릿지 역할만 수행합니다.
